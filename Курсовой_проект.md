# Курсовой проект: ритм-игра для обучения игре на фортепиано

Гибрид **Synthesia** (библиотека MIDI-файлов с падающими нотами) и **Osu!** (оценка точности).
Главный критерий оценки — **база данных** должна быть активным компонентом системы, а не «копилкой».

Этапы: 1) бизнес-логика → 2) база данных на SQLite → 3) реализация в Unity (C#).

---

## Этап 1. Бизнес-логика

### 1.1 Предметная область

- **Объект** — процесс обучения игре на фортепиано / тренировки ритма с помощью MIDI-файлов.
- **Цель системы** — предоставить выбор MIDI-композиции, сыграть её в виде «падающих нот», оценить точность и сохранить историю результатов.

**Границы системы:**

- Входные данные: MIDI-файлы + действия пользователя (нажатия клавиш во времени).
- Выходные данные: точность, сохранённый результат, статистика.
- Стейкхолдеры: студент (пользователь), преподаватель (проверяющий архитектуру).

### 1.2 Бизнес-процесс (сценарий)

1. Загрузить MIDI-файлы в библиотеку.
2. Выбрать композицию по критериям (жанр, сложность, BPM, тональность).
3. Запустить игровой движок:
   - Парсинг MIDI → создание нотной дорожки.
   - Отображение падающих нот.
   - Сбор нажатий клавиш.
   - Сравнение времени нажатия с ожидаемым.
4. Вычислить точность (как в Osu! — 300, 100, 50, miss).
5. Сохранить результат (композиция, точность, дата).
6. Показать прогресс по сложности / инструменту.

Процесс — основа **event-driven модели** (события: файл выбран, нота упала, нажата клавиша, игра закончена).

### 1.3 Модель точности и игрового цикла

**Математическая модель точности:**

- `t_n` — идеальное время нажатия ноты из MIDI.
- `t_u` — фактическое время нажатия пользователем.
- `Δ = |t_u − t_n|`

Зоны оценки (как в Osu!):

| Оценка | Условие | Очки |
|--------|---------|------|
| Perfect | Δ ≤ 50 мс | 300 |
| Good | 50 < Δ ≤ 100 мс | 100 |
| Bad | 100 < Δ ≤ 200 мс | 50 |
| Miss | Δ > 200 мс или нет нажатия | 0 |

**Общая точность** = (сумма очков) / (максимум очков) × 100 %, где максимум = количество нот × 300.

**Модель очереди нот** (игровой цикл) — каждая нота это объект с полями: MIDI-номер (высота), время появления на дорожке, время ожидаемого нажатия, длительность (для удержания). Это детерминированный входной поток из MIDI.

**Конечный автомат игровой сессии (FSM):**

`Idle → Selecting → Loading → Playing → Evaluating → SavingResult → Idle`

### 1.4 Сценарии использования (Use Cases)

1. **Управление библиотекой MIDI** — добавить/удалить MIDI, автоматически извлечь метаданные (BPM, тональность, инструменты, название).
2. **Фильтрация и поиск** — по жанру (тег), BPM (диапазон), сложности (плотность нот в секунду).
3. **Игра** — запуск уровня, обработка ввода с клавиатуры (или MIDI-клавиатуры), визуальная обратная связь (попадание / осечка).
4. **Сохранение истории результатов** — каждый результат привязан к профилю; прогресс по каждой композиции.
5. **Отображение статистики** — график точности по времени, самые сложные места в песне.

## Этап 2. База данных (SQLite)

**Принцип:** БД — активный компонент. Без неё игра не работает:

- MIDI-файлы не читаются напрямую — сначала парсятся и сохраняются в БД (таблица `Notes`). Игра читает ноты **только из БД**.
- Каждое нажатие пишется в `HitEvents` (миллисекунды, точность, палец).
- Динамическая сложность — БД вычисляет: если игрок набирает > 90 % три раза подряд → предлагать +10 % BPM.
- Рекомендации — SQL находит ноты, где чаще всего miss → «Тренируй ноту F4».

### 2.1 Основная схема (3НФ)

```sql
-- 1. ПОЛЬЗОВАТЕЛИ
CREATE TABLE Users (
    user_id INTEGER PRIMARY KEY AUTOINCREMENT,
    username TEXT UNIQUE NOT NULL,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    total_score REAL DEFAULT 0,      -- денормализовано для скорости
    games_played INTEGER DEFAULT 0
);

-- 2. MIDI-ТРЕКИ (метаданные)
CREATE TABLE MidiTracks (
    track_id INTEGER PRIMARY KEY AUTOINCREMENT,
    filename TEXT UNIQUE,
    title TEXT NOT NULL,
    composer TEXT,
    bpm REAL NOT NULL,
    tonality TEXT,                    -- C, G, Am и т.д.
    difficulty REAL,                  -- 1.0 - 10.0 (вычисляется)
    note_density REAL                 -- нот в секунду
);

-- 3. НОТЫ (игра читает отсюда!)
CREATE TABLE Notes (
    note_id INTEGER PRIMARY KEY AUTOINCREMENT,
    track_id INTEGER NOT NULL,
    note_index INTEGER,               -- порядковый номер
    midi_number INTEGER NOT NULL,     -- 60 = C4
    start_time REAL NOT NULL,         -- в секундах от начала
    duration REAL,                    -- для удержания
    hand TEXT CHECK(hand IN ('left','right')),
    finger INTEGER CHECK(finger BETWEEN 1 AND 5),
    FOREIGN KEY (track_id) REFERENCES MidiTracks(track_id) ON DELETE CASCADE
);

-- 4. РЕЗУЛЬТАТЫ ИГРЫ (сессия)
CREATE TABLE GameResults (
    result_id INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id INTEGER NOT NULL,
    track_id INTEGER NOT NULL,
    played_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    accuracy_percent REAL,            -- 0-100
    score_300 INTEGER,
    score_100 INTEGER,
    score_50 INTEGER,
    miss_count INTEGER,
    max_combo INTEGER,
    FOREIGN KEY (user_id) REFERENCES Users(user_id),
    FOREIGN KEY (track_id) REFERENCES MidiTracks(track_id)
);

-- 5. КАЖДОЕ НАЖАТИЕ (данные для анализа)
CREATE TABLE HitEvents (
    hit_id INTEGER PRIMARY KEY AUTOINCREMENT,
    result_id INTEGER NOT NULL,
    note_id INTEGER NOT NULL,
    expected_time REAL,
    actual_time REAL,
    delta_ms INTEGER,                  -- отрицательное = рано
    judgement TEXT CHECK(judgement IN ('perfect','good','bad','miss')),
    finger_used INTEGER,
    FOREIGN KEY (result_id) REFERENCES GameResults(result_id),
    FOREIGN KEY (note_id) REFERENCES Notes(note_id)
);

-- 6. НАСТРОЙКИ ПОЛЬЗОВАТЕЛЯ (влияют на геймплей)
CREATE TABLE UserSettings (
    user_id INTEGER PRIMARY KEY,
    visual_offset_ms INTEGER DEFAULT 0,   -- смещение нот
    audio_offset_ms INTEGER DEFAULT 0,
    min_bpm_filter INTEGER,
    max_bpm_filter INTEGER,
    preferred_hand TEXT DEFAULT 'auto',   -- left/right/auto
    difficulty_filter_min REAL DEFAULT 1,
    difficulty_filter_max REAL DEFAULT 10,
    FOREIGN KEY (user_id) REFERENCES Users(user_id)
);

-- 7. АППЛИКАТУРА (рекомендации для пальцев)
CREATE TABLE FingerMapping (
    mapping_id INTEGER PRIMARY KEY AUTOINCREMENT,
    track_id INTEGER NOT NULL,
    note_id INTEGER NOT NULL,
    recommended_hand TEXT,
    recommended_finger INTEGER,
    FOREIGN KEY (track_id) REFERENCES MidiTracks(track_id),
    FOREIGN KEY (note_id) REFERENCES Notes(note_id)
);
```

### 2.2 Размерность клавиатуры и транспозиция

Решает проблему разных устройств (25 / 49 / 88 клавиш) и возможность поднять/опустить произведение на октаву.

| Проблема | Решение через БД |
|----------|------------------|
| У пользователя 2 октавы, а в MIDI 88 клавиш | Транспонирование (сдвиг нот) + октавный фильтр |
| Нужно поднять/опустить всё произведение | `TrackTransposition` — настройки для каждого трека |
| Разные устройства | `DeviceProfiles` — профили устройств |
| Трек требует ноту, которой нет на устройстве | Автоматический ремаппинг в доступную октаву |

```sql
-- ПРОФИЛИ УСТРОЙСТВ (клавиатуры)
CREATE TABLE DeviceProfiles (
    device_id INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id INTEGER NOT NULL,
    device_name TEXT NOT NULL,           -- "Ноутбук 25 клавиш", "MIDI 88"
    lowest_note INTEGER NOT NULL,        -- MIDI номер самой низкой ноты (36 = C2)
    highest_note INTEGER NOT NULL,
    total_keys INTEGER NOT NULL,
    is_default BOOLEAN DEFAULT 0,
    FOREIGN KEY (user_id) REFERENCES Users(user_id)
);

-- НАСТРОЙКИ ТРАНСПОЗИЦИИ ДЛЯ ТРЕКА
CREATE TABLE TrackTransposition (
    transposition_id INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id INTEGER NOT NULL,
    track_id INTEGER NOT NULL,
    device_id INTEGER NOT NULL,
    transpose_semitones INTEGER DEFAULT 0,   -- -12 = -1 октава, +12 = +1 октава
    auto_transpose BOOLEAN DEFAULT 1,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    last_used DATETIME,
    UNIQUE(user_id, track_id, device_id),
    FOREIGN KEY (user_id) REFERENCES Users(user_id),
    FOREIGN KEY (track_id) REFERENCES MidiTracks(track_id),
    FOREIGN KEY (device_id) REFERENCES DeviceProfiles(device_id)
);

-- ПРАВИЛА РЕМАППИНГА
CREATE TABLE RemappingRules (
    rule_id INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id INTEGER NOT NULL,
    device_id INTEGER NOT NULL,
    original_note INTEGER NOT NULL,
    remapped_note INTEGER NOT NULL,
    is_active BOOLEAN DEFAULT 1,
    FOREIGN KEY (user_id) REFERENCES Users(user_id),
    FOREIGN KEY (device_id) REFERENCES DeviceProfiles(device_id)
);

-- ИСТОРИЯ ТРАНСПОЗИЦИЙ
CREATE TABLE TranspositionHistory (
    history_id INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id INTEGER NOT NULL,
    track_id INTEGER NOT NULL,
    device_id INTEGER NOT NULL,
    transpose_used INTEGER NOT NULL,
    was_auto BOOLEAN,
    timestamp DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES Users(user_id)
);
```

**Минимальные требования трека** (расширение `MidiTracks`):

```sql
ALTER TABLE MidiTracks ADD COLUMN min_required_keys INTEGER;
ALTER TABLE MidiTracks ADD COLUMN recommended_octave_range TEXT; -- "C3-C6"
ALTER TABLE MidiTracks ADD COLUMN can_transpose BOOLEAN DEFAULT 1;
```

**Алгоритм транспозиции:**

1. При выборе трека — проверить диапазон нот (`MIN`/`MAX` midi_number) и сравнить с диапазоном устройства.
2. Если не помещается — вычислить нужный сдвиг с учётом центра клавиатуры.
3. Сохранить сдвиг в `TrackTransposition` (`INSERT ... ON CONFLICT DO UPDATE`).
4. При загрузке нот — применить сдвиг: `n.midi_number + COALESCE(tt.transpose_semitones, 0)`.
5. Если виртуальная нота всё ещё вне диапазона — найти правило в `RemappingRules` (иначе пропустить или сдвинуть в ближайшую октаву).

### 2.4 Сложные SQL-запросы (для защиты)

```sql
-- Запрос 1: Прогресс игрока по треку (сравнение результатов, оконная функция)
SELECT
    r.played_at,
    r.accuracy_percent,
    LAG(r.accuracy_percent) OVER (PARTITION BY r.track_id ORDER BY r.played_at) AS prev_accuracy,
    (r.accuracy_percent - LAG(r.accuracy_percent) OVER (PARTITION BY r.track_id ORDER BY r.played_at)) AS improvement
FROM GameResults r
WHERE r.user_id = 1 AND r.track_id = 5;

-- Запрос 2: Самые сложные ноты (где чаще всего miss)
SELECT n.midi_number, COUNT(*) AS miss_count, n.hand, n.finger
FROM HitEvents h
JOIN Notes n ON h.note_id = n.note_id
WHERE h.judgement = 'miss'
GROUP BY n.midi_number, n.hand, n.finger
ORDER BY miss_count DESC
LIMIT 10;

-- Запрос 3: Рекомендация треков (давно не играл + подходит под уровень)
SELECT t.title, t.difficulty, t.bpm
FROM MidiTracks t
WHERE t.track_id NOT IN (
    SELECT track_id FROM GameResults
    WHERE user_id = 1 AND played_at > date('now', '-7 days')
)
AND t.difficulty BETWEEN
    (SELECT COALESCE(AVG(accuracy_percent)/10, 5) FROM GameResults WHERE user_id = 1)
    AND 10
ORDER BY t.difficulty;

-- Запрос 4: Какая рука/палец опаздывает чаще
SELECT n.hand, n.finger, AVG(h.delta_ms) AS avg_lateness, COUNT(*) AS hit_count
FROM HitEvents h
JOIN Notes n ON h.note_id = n.note_id
WHERE h.judgement IN ('good','bad')
GROUP BY n.hand, n.finger
ORDER BY avg_lateness DESC;
```

### 2.6 Триггеры

```sql
-- При вставке результата обновляем статистику пользователя
CREATE TRIGGER update_user_stats AFTER INSERT ON GameResults
BEGIN
    UPDATE Users SET
        total_score = (SELECT AVG(accuracy_percent) FROM GameResults WHERE user_id = NEW.user_id),
        games_played = games_played + 1
    WHERE user_id = NEW.user_id;
END;

-- Автоматический расчёт диапазона трека при добавлении нот
CREATE TRIGGER update_track_range AFTER INSERT ON Notes
BEGIN
    UPDATE MidiTracks SET
        min_required_keys = (SELECT MAX(midi_number) - MIN(midi_number) FROM Notes WHERE track_id = NEW.track_id),
        recommended_octave_range = (
            SELECT 'C' || ((MIN(midi_number) - 12) / 12) || '-C' || ((MAX(midi_number) - 12) / 12)
            FROM Notes WHERE track_id = NEW.track_id
        )
    WHERE track_id = NEW.track_id;
END;
```

### 3.1 Импорт MIDI-файла

```csharp
using NAudio.Midi;
using System.IO;

public class MidiImporter : MonoBehaviour
{
    public void ImportMidiFile(string filePath)
    {
        var midiFile = new MidiFile(filePath);

        int trackId = Database.InsertTrack(
            Path.GetFileName(filePath),
            DetectBPM(midiFile),
            DetectTonality(midiFile)
        );

        var notes = ExtractNotes(midiFile);
        foreach (var note in notes)
            Database.InsertNote(trackId, note);

        AutoAssignFingers(trackId);   // простой алгоритм по умолчанию
    }

    private List<NoteData> ExtractNotes(MidiFile midiFile)
    {
        var notes = new List<NoteData>();
        var track = midiFile.Events[0]; // обычно мелодия в первом треке

        foreach (var ev in track)
        {
            if (ev is NoteOnEvent noteOn)
            {
                notes.Add(new NoteData
                {
                    midiNumber = noteOn.NoteNumber,
                    startTime  = noteOn.AbsoluteTime / 100.0,
                    duration   = noteOn.NoteLength / 100.0
                });
            }
        }
        return notes;
    }
}
```

## SOLID-архитектура

Принципы заложены, чтобы расширять систему без переписывания (на дипломе). **Заглушки в курсовой не показываем** — только то, что реально работает; интерфейсы оставляем как обоснование расширяемости.

| Принцип | Применение |
|---------|------------|
| Single Responsibility | Каждый модуль отвечает за одно |
| Open/Closed | БД и интерфейсы открыты для расширения (новые типы нот, режимы) |
| Liskov Substitution | Все редакторы работают через `INoteEditor` |
| Interface Segregation | Мелкие интерфейсы вместо одного «толстого» |
| Dependency Inversion | Модули зависят от абстракций БД, не наоборот (SQLite → можно заменить) |

**Слои:** Презентация (Unity Views) → Бизнес-логика (`GameEngine`, `INoteEditor`, `StatsAnalyzer`) → Доступ к данным (репозитории-интерфейсы → `SQLiteRepo`) → БД (SQLite).

## Итоговый объём курсовой

**Делаем:**

- ✅ SQLite БД (главный фокус) — таблицы, индексы, триггеры, сложные запросы.
- ✅ Загрузка своих MIDI-файлов (парсинг → БД).
- ✅ Выбор трека с сортировкой/фильтром по BPM и сложности.
- ✅ Игровой движок (падающие ноты, нажатия, оценка).
- ✅ Транспозиция и профили устройств (25 / 49 / 88 клавиш).
- ✅ Редактирование пальцев (вручную + шаблоны).
- ✅ Сохранение результатов и статистики.
- ✅ SOLID-архитектура с интерфейсами.

**Не делаем:**

- ❌ Визуальный редактор нот (пиано-ролл).
- ❌ Создание MIDI с нуля.
- ❌ Онлайн-таблицы лидеров.

**Список таблиц БД:** Users, MidiTracks, Notes, GameResults, HitEvents, UserSettings, FingerMapping, DeviceProfiles, TrackTransposition, RemappingRules, TranspositionHistory, UserFingerAssignments, FingerTemplates, TemplateNoteMapping, SchemaVersion.
